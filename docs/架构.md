# fuelConsumption 项目架构

## 项目定位

fuelConsumption 是一个离线优先的 Flutter 能耗与费用记录应用，用于管理多辆车的加油、充电、插混补能和保养费用记录。当前实现重点是本地记录、油耗/费用统计、趋势展示、车辆管理，以及通过 JSON 手动导入导出备份数据。

应用目前以 Android 验证为主。核心运行数据存储在本机 SQLite 数据库中，不依赖远程服务。

## 技术栈

- Flutter / Dart：应用框架和语言。
- Material 3：主要 UI 组件体系。
- flutter_riverpod：托管 repository 生命周期，并保留后续迁移页面状态和 stream 的空间。
- drift + sqlite3_flutter_libs：本地 SQLite 持久化和类型安全查询。
- path_provider + path：定位应用文档目录并创建数据库文件路径。
- fl_chart：仪表盘趋势图。
- collection：领域统计中的集合排序辅助。
- uuid：车辆和记录 ID 生成。
- build_runner + drift_dev：生成 Drift 数据库代码。

## 分层职责

### 入口层

`lib/main.dart` 负责 Flutter 初始化和启动装配：在根 `ProviderScope` 中把 presentation 层的 `appRepositoryProvider` 绑定到 data 层 `localRepositoryProvider`。真实数据库由 `localRepositoryProvider` 创建，并在 provider dispose 时关闭数据库连接。`FuelConsumptionApp` 只负责主题和根 UI。

### UI 层

- `FuelConsumptionApp`：配置主题和根页面。
- `presentation/app_providers.dart`：集中放置 Riverpod provider，把 repository 端口、Dashboard 查询流和写侧 command 接线到 UI；provider 不进入 application 业务服务。
- `presentation/dashboard_navigation.dart`：集中定义 Dashboard 页面、底部 tab 和“记一笔”动作，供 screens 和 widgets 共同依赖，避免 widgets 反向依赖 screens。
- `DashboardScreen`：负责页面脚手架、弹窗展示和把用户动作转发给 application command。
- `DashboardController`：管理底部 tab、当前页面、车辆选择和记一笔入口的 UI 状态转换。
- `dashboard_dialogs.dart`：集中构建 Dashboard 使用的创建菜单、车辆弹窗、删除确认、备份导入和 JSON 展示弹窗，避免主屏继续膨胀。
- `widgets/dashboard/*`：Dashboard 加载和错误等纯展示状态。
- `ConsumptionScreen`：油耗首页编排，车辆状态、首页费用、统计卡片和优惠横幅由 `widgets/consumption/*` 承担。
- `ExpenseScreen`：费用页编排，概览、指标、构成、年度统计和明细由 `widgets/expense/*` 承担。
- `ConsumptionTrendCard` / `MonthlyFuelCostCard` / `AnnualConsumptionCard`：首页图表组件。
- `AnnualExpenseCard` / `ExpenseDetailCard`：费用页年度图表、明细和结构化燃油优惠展示组件。
- `EnergyRecordScreen`：补能入口路由，根据车辆类型分派到燃油、充电或油电表单。
- `RefuelScreen`：燃油/摩托车优惠加油表单页面，负责日期、选项、保存调度和错误展示。
- `ChargeScreen`：电车充电表单页面，保存 kWh、充电单价和快慢充模式。
- `HybridScreen`：插混油电补能表单页面，保存燃油和电量两组数量、单价和总费用。
- `RefuelFormController`：管理加油页文本控制器和机显金额、优惠、实付金额联动。
- `widgets/refuel/*`：加油页日期、里程、金额、分段开关和选填项表单组件。
- `MaintenanceScreen`：保养/维修等非能源费用表单。
- `MineScreen`：车辆管理、本地备份、帮助、反馈和隐私说明。
- `widgets/mine/*`：我的中心的分组卡片、菜单项、车辆行、说明弹层和反馈弹层。
- `CreateRecordSheet`：底部“记一笔”根据车辆类型展示加油、充电或油电补能，并可选择保养。
- `VehicleDialog`、`ImportDialog`、`TextPayloadDialog`：车辆输入、备份导入交互和 JSON 展示弹窗；车辆保存和导入执行通过 callback/action 注入。

当前 `ConsumptionScreen` 和 `ExpenseScreen` 已收敛为页面编排，主要卡片迁移到 `widgets/consumption/*` 和 `widgets/expense/*`。`EnergyRecordScreen` 负责按车辆类型选择补能表单；`RefuelScreen` 的金额联动和文本控制器已迁移到 `RefuelFormController`。燃油、充电、油电和保养表单只收集文本、日期和选项，并提交 `RefuelRecordInput` / `ChargeRecordInput` / `HybridRecordInput` / `MaintenanceRecordInput`；ID 生成、领域组装、校验和保存由 application command 负责。`ChargeScreen` 和 `HybridScreen` 保持独立页面文件，避免把电车/插混字段堆回燃油表单。

### 应用层

- `application/backup/*`：定义 `BackupData`、备份 schema、JSON 编解码和领域对象与备份 map 的转换；旧备份字段兼容集中在这里。
- `BackupImportService`：负责 JSON 解析、备份语义校验、导入前自动备份和导入执行。
- `DashboardCommandService`：封装 Dashboard 写侧入口，统一接线车辆创建/删除、补能保存、保养保存、备份导出和导入 action 创建；记录保存委派给 `RecordCommandService`。
- `RecordCommandService`：接收表单提交的 `*RecordInput`，生成记录 ID，调用领域 assembler 组装 `EnergyRecord`，执行里程/金额校验，并通过 repository 保存补能和保养记录。
- `DashboardLoadState` / `resolveDashboardVehicle`：定义 Dashboard 读模型和车辆选择规则；Riverpod 监听由 presentation provider 承担。
- `DashboardData`：封装 Dashboard 页面所需的选中车辆、记录排序、保养记录和统计快照。
- `VehicleCommandService`：负责把车辆表单 draft 转换为带 ID 的 `Vehicle` 并调用 repository 保存。
- `application/ports/AppRepository`：定义应用层依赖的数据访问端口；data 层提供当前 SQLite 实现。
- `ImportDialogActions` / `VehicleDraft` / `*RecordInput`：作为 UI 和应用服务之间的轻量边界，避免 widget 直接依赖 repository、codec、UUID 生成或领域组装细节。

application 层不依赖 Flutter、Riverpod、data、screens 或 widgets；domain 不依赖框架和基础设施；presentation、screens、widgets 不直连 data；widgets 不反向依赖 screens。这些边界由 `test/architecture/layer_boundaries_test.dart` 覆盖。

### 领域层

`lib/src/domain/models.dart` 定义核心业务对象：

- `VehicleType`：燃油车、电车、插混、摩托车。
- `EnergyType`：加油、充电、油电。
- `ChargeMode`：慢充、快充。
- `Vehicle`：车辆基础信息。
- `EnergyRecord`：补能记录，覆盖燃油、充电和插混。
- `MaintenanceRecord`：保养/维修等非能源费用记录。
- `StatisticsSnapshot`：UI 展示用统计结果。

领域计算分布如下：

- `statistics.dart`：计算平均能耗、最近能耗、总费用、每公里成本和累计里程。
- `consumption_statistics.dart`：首页费用与优惠统计。
- `expense_statistics.dart`：费用页汇总、年度聚合和明细列表。
- `fuel_grades.dart`：燃油标号常量。
- `legacy_refuel_note_parser.dart`：只负责旧备注金额协议解析和展示备注清理，用于数据库迁移、旧备份导入和兼容兜底。
- `record_amounts.dart`：优先读取补能记录中的结构化机显金额、优惠和实付金额；旧备注只作为历史兼容来源。
- `refuel_amount_calculator.dart`：加油页机显金额、优惠和实付金额联动计算。
- `refuel_record_assembler.dart`：将加油表单草稿组装为 `EnergyRecord`，并返回表单级错误。
- `energy_record_assembler.dart`：将充电和油电表单草稿组装为 `EnergyRecord`，并返回表单级错误。
- `validation.dart`：补能记录和保养记录校验。

### 数据层

`AppRepository` 位于 application ports，定义 UI 和应用层依赖的数据访问抽象，覆盖车辆、补能记录、保养记录、备份导入导出和备份校验。data 层的 `localRepositoryProvider` 负责创建 `AppDatabase` 和 `FuelRepository`，并由根 `ProviderScope` 注入到 presentation 层的 `appRepositoryProvider`。

`FuelRepository` 是当前实现：

- 将读写委托给 `AppDatabase`。
- 保存车辆、补能记录和保养记录前执行领域校验。
- 导出时组装 `BackupData`。
- 导入前通过 `BackupValidator` 校验 ID 唯一性、车辆引用和记录合法性。
- 校验通过后全量替换数据库内容。

`AppDatabase` 暴露数据库读写 API，具体职责拆分为：

- `app_database_tables.dart`：`VehicleRows`、`EnergyRecordRows`、`MaintenanceRecordRows` 表定义。
- `app_database_migrations.dart`：schema v2/v3/v4 迁移、v4 历史燃油优惠回填和 foreign key 开启。
- `app_database_mappers.dart`：row/domain companion 和实体映射；数据库 row 到领域对象的映射在 data 层显式完成，不复用 application backup JSON serializer。
- `app_database_connection.dart`：真实 SQLite 连接和内存测试连接。
- `app_database_vehicle_dao.dart`：车辆写入、删除和车辆列表查询。
- `app_database_record_dao.dart`：补能记录写入和查询。
- `app_database_maintenance_dao.dart`：保养记录写入和查询。
- `app_database_backup_dao.dart`：导入备份时的整库替换事务。

数据库文件位于应用文档目录，文件名为 `fuel_consumption.sqlite`。数据库 schema version 当前为 `4`：v2 追加车型/单价字段，v3 新增保养记录表，v4 新增机显金额、实付金额和优惠金额结构化字段，并从历史备注中回填旧加油记录的结构化金额。`app_database.g.dart` 是 Drift 生成文件，不应手动修改。

### Android 发布层

- `android/app/build.gradle.kts`：Android namespace 和 applicationId 均为 `com.fuelconsumption.app`，release 签名从本机 `android/key.properties` 读取。
- `android/app/src/main/AndroidManifest.xml`：应用显示名通过 `@string/app_name` 管理，启动图标使用 `@mipmap/ic_launcher`。
- `android/app/src/main/res/values/strings.xml`：当前应用名为“油耗”。
- `android/app/src/main/kotlin/com/fuelconsumption/app/MainActivity.kt`：Android 启动 Activity 已迁移到正式包路径。

## 运行时数据流

```text
main.dart
  -> FuelConsumptionApp
  -> appRepositoryProvider override(localRepositoryProvider)
  -> DashboardScreen
  -> AppRepository
  -> FuelRepository
  -> AppDatabase
  -> SQLite
```

展示数据流：

1. presentation 层 `vehiclesProvider` 监听车辆列表。
2. presentation 层 `dashboardQueryProvider` 解析当前车辆后监听该车补能记录和保养记录。
3. `DashboardLoadState` 表示 loading、empty、ready、error，并保留可用车辆上下文。
4. `DashboardScreen` 根据 `DashboardLoadState` 渲染油耗、费用、补能、保养或我的中心页面。

写入数据流：

1. 用户通过底部表单创建车辆、补能记录或保养记录。
2. UI 产出 `VehicleDraft` 或 `*RecordInput`，并把动作交给 application command。
3. `DashboardCommandService` 转发记录保存到 `RecordCommandService`；车辆创建交给 `VehicleCommandService`。
4. `RecordCommandService` 生成 ID、调用领域 assembler 组装记录、读取历史记录并执行校验，然后调用 `AppRepository`。
5. `FuelRepository` 执行仓储层兜底校验。
6. `AppDatabase` 写入 Drift/SQLite。
7. Drift stream 推送变化，界面自动刷新。

备份数据流：

1. 导出时，`FuelRepository.exportBackup()` 读取车辆、补能记录和保养记录。
2. `BackupCodec.encode()` 将 `BackupData` 转为 JSON。
3. 导入时，`ImportDialog` 将输入交给 `BackupImportService.parseAndValidate()`，拿到可预览的 `BackupData`；旧备份中缺失的燃油优惠结构化字段会在 `backup_serializers.dart` 中从历史备注派生。
4. 用户确认替换后，`BackupImportService.importBackup()` 先导出现有数据，再调用 `FuelRepository.importBackup()`。
5. `FuelRepository.importBackup()` 使用 `BackupValidator` 再次校验并调用 `AppDatabase.replaceAll()`。
