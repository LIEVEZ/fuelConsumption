# fuelConsumption 项目架构

## 项目定位

fuelConsumption 是一个离线优先的 Flutter 能耗与费用记录应用，用于管理多辆车的加油、充电、插混补能和保养费用记录。当前实现重点是本地记录、油耗/费用统计、趋势展示、车辆管理，以及通过 JSON 手动导入导出备份数据。

应用目前以 Android 验证为主。核心运行数据存储在本机 SQLite 数据库中，不依赖远程服务。

## 技术栈

- Flutter / Dart：应用框架和语言。
- Material 3：主要 UI 组件体系。
- flutter_riverpod：托管 repository 生命周期，并保留后续迁移页面状态和 stream 的空间。
- drift + sqlite3_flutter_libs：本地 SQLite 持久化和类型安全查询。
- path_provider + path：定位应用文档目录并创建数据库文件路径。
- fl_chart：仪表盘趋势图。
- collection：领域统计中的集合排序辅助。
- uuid：车辆和记录 ID 生成。
- build_runner + drift_dev：生成 Drift 数据库代码。

## 分层职责

### 入口层

`lib/main.dart` 只负责 Flutter 初始化并启动 `FuelConsumptionApp`。真实数据库和 repository 由 `repositoryProvider` 创建并在 provider dispose 时关闭数据库连接。

### UI 层

- `FuelConsumptionApp`：配置 `ProviderScope`、主题和根页面。
- `DashboardScreen`：负责底部 tab、车辆选择、记一笔入口、导入导出调度。
- `DashboardDataBuilder`：监听车辆、补能记录和保养记录，组装页面需要的数据，并处理空、加载和错误状态。
- `ConsumptionScreen`：油耗首页、车辆概览、能耗统计和趋势。
- `ExpenseScreen`：费用概览、年度统计和费用明细。
- `ConsumptionTrendCard` / `MonthlyFuelCostCard` / `AnnualConsumptionCard`：首页图表组件。
- `AnnualExpenseCard` / `ExpenseDetailCard`：费用页年度图表和明细组件。
- `RefuelScreen`：优惠加油表单，支持机显金额、优惠和实付金额联动。
- `widgets/refuel/*`：加油页日期、里程、金额、分段开关和选填项表单组件。
- `MaintenanceScreen`：保养/维修等非能源费用表单。
- `MineScreen`：车辆管理、本地备份、帮助、反馈和隐私说明。
- `widgets/mine/*`：我的中心的分组卡片、菜单项、车辆行、说明弹层和反馈弹层。
- `CreateRecordSheet`：底部“记一笔”选择加油或保养。
- `VehicleDialog`、`ImportDialog`、`TextPayloadDialog`：车辆、新增备份导入和 JSON 展示弹窗。

当前 `ConsumptionScreen`、`ExpenseScreen` 和 `RefuelScreen` 已拆出部分图表、明细和表单逻辑；后续可以继续按“页面、表单状态、卡片、图表、列表项”拆分。

### 领域层

`lib/src/domain/models.dart` 定义核心业务对象：

- `VehicleType`：燃油车、电车、插混、摩托车。
- `EnergyType`：加油、充电、油电。
- `ChargeMode`：慢充、快充。
- `Vehicle`：车辆基础信息。
- `EnergyRecord`：补能记录，覆盖燃油、充电和插混。
- `MaintenanceRecord`：保养/维修等非能源费用记录。
- `StatisticsSnapshot`：UI 展示用统计结果。
- `BackupData`：导入导出的备份数据包。

领域计算分布如下：

- `statistics.dart`：计算平均能耗、最近能耗、总费用、每公里成本和累计里程。
- `consumption_statistics.dart`：首页费用与优惠统计。
- `expense_statistics.dart`：费用页汇总、年度聚合和明细列表。
- `fuel_grades.dart`：燃油标号常量。
- `record_amounts.dart`：从现有记录备注中解析机显金额、优惠和实付金额。
- `refuel_amount_calculator.dart`：加油页机显金额、优惠和实付金额联动计算。
- `refuel_record_assembler.dart`：将加油表单草稿组装为 `EnergyRecord`，并返回表单级错误。
- `validation.dart`：补能记录和保养记录校验。

### 数据层

`AppRepository` 定义 UI 依赖的数据访问抽象，覆盖车辆、补能记录、保养记录、备份导入导出和备份校验。

`FuelRepository` 是当前实现：

- 将读写委托给 `AppDatabase`。
- 保存车辆、补能记录和保养记录前执行领域校验。
- 导出时组装 `BackupData`。
- 导入前校验 ID 唯一性、车辆引用和记录合法性。
- 校验通过后全量替换数据库内容。

`AppDatabase` 定义 Drift 表和数据库操作：

- `VehicleRows`：车辆表。
- `EnergyRecordRows`：补能记录表。
- `MaintenanceRecordRows`：保养记录表。
- `watchVehicles()` / `watchRecords()` / `watchMaintenanceRecords()`：stream 监听。
- `getVehicles()` / `getRecords()` / `getMaintenanceRecords()`：一次性读取。
- `upsertVehicle()` / `upsertRecord()` / `upsertMaintenanceRecord()`：写入或更新。
- `replaceAll()`：导入备份时清空并重建数据。

数据库文件位于应用文档目录，文件名为 `fuel_consumption.sqlite`。数据库 schema version 当前为 `3`：v2 追加燃油折扣相关字段，v3 新增保养记录表。`app_database.g.dart` 是 Drift 生成文件，不应手动修改。

## 运行时数据流

```text
main.dart
  -> FuelConsumptionApp
  -> repositoryProvider
  -> DashboardScreen
  -> AppRepository
  -> FuelRepository
  -> AppDatabase
  -> SQLite
```

展示数据流：

1. `DashboardDataBuilder` 监听车辆列表。
2. 解析当前车辆后监听该车补能记录和保养记录。
3. 组装 `DashboardData` 并传给油耗、费用、加油、保养或我的中心页面。

写入数据流：

1. 用户通过底部表单创建车辆、补能记录或保养记录。
2. UI 构造 `Vehicle`、`EnergyRecord` 或 `MaintenanceRecord`。
3. `FuelRepository` 执行校验。
4. `AppDatabase` 写入 Drift/SQLite。
5. Drift stream 推送变化，界面自动刷新。

备份数据流：

1. 导出时，`FuelRepository.exportBackup()` 读取车辆、补能记录和保养记录。
2. `BackupCodec.encode()` 将 `BackupData` 转为 JSON。
3. 导入时，`ImportDialog` 解析并预览 JSON。
4. `FuelRepository.validateBackup()` 先做只读校验。
5. 用户确认替换后，`FuelRepository.importBackup()` 再次校验并调用 `AppDatabase.replaceAll()`。

## 当前工程状态

- 核心应用已具备车辆管理、补能记录、保养记录、统计展示、费用页、我的中心和 JSON 手动备份能力。
- 底部菜单为油耗、费用、记一笔、优惠加油、我的；保养通过“记一笔”进入。
- 应用已接入统一天蓝色主题。
- 仓储生命周期已收口到 Riverpod provider，页面数据流仍保留显式 repository 和 StreamBuilder，便于后续渐进迁移。
- 当前仓库已包含领域、数据和 widget 测试，覆盖统计、校验、加油金额联动、加油记录组装、加油页基础保存流程、备份错误提示、导入预览、记一笔弹层和我的中心基础展示。
- 当前仓库已包含 GitHub Actions CI，覆盖 `flutter pub get`、`flutter analyze`、`flutter test --concurrency 1`。
- Android 包名仍是模板式 `com.example.fuel_consumption`，正式发布前需要替换为真实唯一包名。
- Android release 当前使用 debug 签名配置，只适合开发验证，不适合正式发布。
